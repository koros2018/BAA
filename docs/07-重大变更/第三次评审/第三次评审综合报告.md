## 二、交叉穿透分析：API需求对架构的影响

### 2.1 关键发现

EMA2 API需求（06-16）是一个**重大新增约束**，它改变了BAA的交付形态：

**之前（前两次评审）：** BAA是一个独立软件产品 → 前端Web界面+后端API
**现在（加入API需求后）：** BAA是一个**服务化产品** → 核心能力需封装为3个HTTP接口，同时保留独立Web界面

### 2.2 对架构的影响

```
之前的架构（独立软件视角）：
用户 → Web UI → BAA后端 → 图纸解析+原子函数判定 → 审查报告
                    │
                    └── 本地调用

加入API需求后的架构（服务化视角）：
EMA2前端 → EMA2后端(7188) → BAA Client → BAA API服务
                                            ↓
                                      图纸解析+原子函数判定
                                            ↓
                                      审查报告/MODEL文件

保留：Web UI → BAA后端（直接模式，不经过EMA2）
```

### 2.3 架构调整方案

BAA后端需要同时支持两种调用方式：

```
┌─────────────────────────────────────────────────────────┐
│              BAA API 服务层（新增——P0）                    │
├─────────────────────────────────────────────────────────┤
│  POST /deconstruct    图纸解构（免费）                    │
│  POST /reconstruct    BIM重构（收费）                    │
│  GET  /order/{id}     订单状态查询（可选）                │
│  ─── 认证: Bearer <REDACTED>                            │
│  ─── 文件上传: multipart/form-data                       │
└────────────────────────┬────────────────────────────────┘
                         │ 内部调用
┌────────────────────────▼────────────────────────────────┐
│            BAA 核心引擎（保持不变）                        │
│  图纸解析 → 语义识别 → 原子函数判定 → 归因分析 → 审查报告  │
└─────────────────────────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│            BAA Web UI（保留，独立部署）                   │
│  系统概览 | 图纸管理 | 规范库 | AI审图 | 对比 | 结果分析  │
└─────────────────────────────────────────────────────────┘
```

**结论：** API服务层是BAA核心引擎的"外壳"，核心逻辑不用变，只需新增3个入口端点+认证机制。估算工作量：**2-3天**（基于FastAPI已有经验）。

### 2.4 API需求对MVP范围的修正

EMA2的API需求要求BAA能：
1. 接收图纸文件 → 返回解构结果（elements列表）
2. 接收file_id+elements → 返回BIM模型文件（目前mock）

**这对MVP意味着：**

| 维度 | 此前MVP | 加入API需求后 | 影响 |
|------|---------|-------------|------|
| 图纸解析 | 仅本地Web上传 | 还需支持API文件上传 | 新增multipart处理，0.5天 |
| 输出格式 | 审查报告HTML/JSON | 还需输出elements结构体 | 输出接口对齐，0.5天 |
| 认证 | 无（本地） | 需Bearer <REDACTED> | 新增认证中间件，0.5天 |
| 文件管理 | 本地临时文件 | 需file_id持久化 | 新增文件管理器，1天 |
| BIM模型输出 | 不输出（仅报告） | 需输出model_url | **P2级，MVP可mock** |

**关键判断：** API需求不是新增功能，而是将现有能力**重新暴露为API**。核心代码无变动，MVP工期不受实质影响。BIM重构输出（/reconstruct）在MVP阶段可继续保持mock模式——就像EMA2目前的mock一样。

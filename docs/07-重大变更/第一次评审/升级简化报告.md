# Blueprint-AI Agent 升级简化报告

> **日期：** 2026-06-07
> **作者：** 第三实例 🔱
> **基于资料：** DeepSeek对话、千问对话、人脸表情识别系统（论文+前端+源码）
> **目标：** 将BAA从"功能堆叠型工具"升级为"AI Agent驱动的图纸智能体"

---

## 一、资料阅读理解总结

### 1.1 DeepSeek对话核心结论

DeepSeek在5轮对话中，逐步将需求从"图纸AI助手"收敛为"图纸合规Agent"：

| 轮次 | 核心观点 |
|------|----------|
| 第1轮 | 完整方案：Electron + React + YOLOv8 + PaddleOCR + Drools + MinIO + PostgreSQL |
| 第2轮 | **关键转折**：聚焦双功能——"图纸解构→合规修正→重构" + "意图解析→概念图→详细图" |
| 第3轮 | **技术路线决策**：放弃AutoCAD，推荐ODA商业路线（年费模式），强烈不推荐纯开源（GPL陷阱+写入不稳定） |
| 第4轮 | **架构定义**：SubAgent001（判别式/RAG+规则引擎）+ SubAgent002（生成式/参数化模板+物理验证） |
| 第5轮 | **市场验证**：品览、中望、品茗、帮图等竞品分析，确认"全自动闭环修正"是市场空白 |

**核心洞察：** "纯大模型搞不定，大模型+传统CAD引擎+规则引擎"的混合架构是唯一可行路线。

### 1.2 千问对话核心结论

千问的补充视角：

- **UI设计**：Windows原生风格（WinUI 3/Fluent Design），左侧导航+顶部标题+中央工作区
- **知识库建设**：规范→JSON三元组（条件-动作-结果），图集→向量数据库+JSON库
- **SubAgent训练**：正向训练SubAgent001（判别），数据验证后训练SubAgent002（生成），形成数据飞轮
- **MCP/Skill vs 独立软件**：推荐"后端微服务+MCP Server"渐进式架构，核心能力封装为MCP工具
- **前端原型**：基于人脸表情识别系统HTML改造的BAA-agent原型（7个模块）

### 1.3 人脸表情识别系统的可借鉴点

虽然领域完全不同，但该系统的工程化思路极具参考价值：

| 维度 | 人脸表情系统 | → BAA-agent |
|------|-------------|---------------------|
| **数据预处理** | 归一化、灰度化、去噪、数据增强 | 图纸矢量化清洗、图层标准化、参数化扰动增强 |
| **模型架构** | CNN基础模块→残差模块→注意力模块→分类输出 | 几何解析模块→规范映射模块→修正生成模块→合规输出 |
| **注意力机制** | 聚焦眉毛/眼睛/嘴巴等关键区域 | 聚焦楼梯间/疏散出口/防火分区等关键区域 |
| **混淆矩阵** | 恐惧vs惊讶易混淆 | 防火墙厚度不足vs防火隔墙厚度不足易混淆 |
| **训练策略** | 早停、学习率衰减、模型检查点 | 迭代上限、修正评分函数、收敛检测 |
| **可视化** | 热力图显示关注区域 | 热力图显示违规区域 |
| **前端架构** | Tailwind + Chart.js + 左侧导航 + 统计卡片 | 直接复用此架构，替换为图纸AI功能模块 |

## 二、当前BAA项目的核心问题

### 2.1 问题诊断

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **功能堆叠，缺乏Agent闭环** | 🔴 严重 | 20个API端点是独立功能点，没有形成"分析→审查→修正→验证"的闭环 |
| **纯规则引擎，没有AI大脑** | 🔴 严重 | review.py是硬编码规则，inference.py是简单统计，没有LLM参与决策 |
| **文档生成是鸡肋** | 🟡 中等 | documents.py生成的文档质量低，实际工程中不可用 |
| **全生命周期模块是空壳** | 🟡 中等 | lifecycle.py只有状态机，没有真正的资料管理和计划联动 |
| **国标库无法独立运行** | 🟡 中等 | specs.py嵌入在主程序中，不能独立部署和扩容 |
| **没有过程资料存储** | 🔴 严重 | 每次分析结果不持久化，无法追溯和审计 |
| **前端UI过时** | 🟢 轻微 | Apple风格UI不符合工程领域用户习惯 |

### 2.2 现有资产盘点（可复用）

| 模块 | 可复用度 | 说明 |
|------|----------|------|
| `core.py` (BlueprintParser) | ✅ 高 | 统一解析接口，PDF/DWG/DXF |
| `dxf_parser.py` + `dwg_extractor.py` | ✅ 高 | 图纸解析核心 |
| `dxf_editor.py` (DXFEditor) | ✅ 高 | DXF编辑，12项操作 |
| `specs.py` (LAYER_TO_SPECS) | ✅ 高 | 国标知识库基础数据 |
| `review.py` | 🟡 中 | 规则引擎基础，需升级为Agent |
| `inference.py` | 🟡 中 | 图纸类型识别，需增强 |
| `budget.py` | 🟢 低 | 可保留但优先级降低 |
| `documents.py` | 🔴 低 | 建议移除或大幅简化 |
| `pdf_documents.py` | 🔴 低 | 建议移除 |
| `optimizer.py` / `completion.py` / `lifecycle.py` | 🟡 中 | 需重写为Agent工作流 |
| `api_server.py` | ✅ 高 | FastAPI框架保留，端点重构 |
| `libredwg-web` (WASM) | ✅ 高 | DWG转换能力 |

### 3.2 核心闭环设计

**闭环一：图纸合规修正（功能一）**

```
输入图纸 → 解析解构 → 结构化数据 → 规范比对 → 发现违规
    ↑                                              │
    │         ┌────────────────────────────────────┘
    │         ▼
    │   生成修正策略(LLM) → 执行修正(CAD引擎) → 重新验证
    │         │                                      │
    │         └──────────── 迭代直到100% ────────────┘
    │                        │
    ▼                        ▼
输出合规图纸 + 完整过程资料（持久存储）
```

**闭环二：意图驱动设计（功能二）**

```
输入建造意图 → 意图解析(LLM) → 结构化需求 → 概念生成
    ↑                                              │
    │         ┌────────────────────────────────────┘
    │         ▼
    │   用户确认/修改 → 详细设计(参数化模板) → 规范验证
    │         │                                      │
    │         └──────────── 迭代直到满意 ────────────┘
    │                        │
    ▼                        ▼
输出正式合规图纸 + 完整过程资料（持久存储）
```

### 3.3 模块简化决策

| 原模块 | 决策 | 理由 |
|--------|------|------|
| `documents.py` | **移除** | 生成的文档质量低，工程中不可用 |
| `pdf_documents.py` | **移除** | 同上 |
| `budget.py` | **降级为工具** | 保留简单估算，不再是独立模块 |
| `optimizer.py` | **合并入SubAgent-002** | 优化就是生成式Agent的一部分 |
| `completion.py` | **重写为过程资料管理** | 竣工资料自动生成，不是独立功能 |
| `lifecycle.py` | **重写为全生命周期资料管理** | 加入计划提醒、资料审核、竣工归档 |
| `review.py` | **升级为SubAgent-001** | 从规则引擎升级为RAG+LLM混合 |
| `specs.py` | **独立为知识库微服务** | 可独立部署、扩容、修正 |
| `dxf_editor.py` | **保留并增强** | 作为SubAgent-002的执行工具 |
| `core.py` + `dxf_parser.py` | **保留** | 作为图纸解析工具层 |
| `inference.py` | **增强** | 加入LLM辅助的图纸语义理解 |

### 3.4 知识库建设方案

这是项目的核心壁垒，分三步走：

**Step 1：规范JSON库**

```json
{
  "id": "GB50016-2016-5.5.18",
  "standard": "GB 50016-2014",
  "clause": "5.5.18",
  "title": "疏散楼梯净宽",
  "condition": "建筑类型=高层公共建筑 AND 楼梯类型=疏散楼梯",
  "requirement": "净宽 >= 1.2m",
  "mandatory": true,
  "category": "消防",
  "check_type": "geometry_comparison",
  "target_entity": "staircase",
  "target_property": "clear_width",
  "operator": ">=",
  "threshold": 1.2,
  "unit": "m"
}
```

**Step 2：图集向量库**

- 选取10-20个典型图集（如12J系列楼梯、22G101钢筋）
- 用ODA SDK提取几何特征
- 向量化后存入ChromaDB
- 支持"以图找图"和"相似构件检索"

**Step 3：过程资料库**

- 每次Agent执行的完整记录
- 包括：原始数据、中间版本、审查意见、修正记录、调用规范条款
- 形成"工程合规纠错数据集"——这是数据飞轮的核心

### 3.5 技术选型

| 层级 | 技术 | 理由 |
|------|------|------|
| **CAD引擎** | ODA Drawings SDK（商业）或 ezdxf（开源过渡） | ODA是商业级稳定性的唯一选择；开发期可用ezdxf过渡 |
| **LLM** | 云端API（DeepSeek/Qwen）+ 本地Ollama（可选） | 云端API能力强，本地Ollama保护数据隐私 |
| **知识库** | PostgreSQL（结构化）+ ChromaDB（向量） | 成熟稳定，社区活跃 |
| **文件存储** | MinIO | S3兼容，可本地部署 |
| **前端** | 复用千问生成的HTML原型，Windows风格 | Tailwind + Chart.js，已验证可用 |
| **后端** | FastAPI（保留） | 已有基础，重构端点即可 |
| **任务调度** | Celery + Redis | 异步任务（AI分析、图纸转换） |
| **Agent框架** | 自研轻量工作流引擎 | 不依赖LangChain等重型框架，保持简洁 |

## 五、风险与应对

| 风险 | 级别 | 应对 |
|------|------|------|
| ODA商业授权成本高 | 🟡 中 | 开发期用ezdxf过渡；收入确认后再采购ODA |
| LLM在工程规范判定中产生幻觉 | 🔴 高 | 规则引擎为主，LLM为辅；所有判定必须引用具体规范条款 |
| 图纸解构的几何保真度不足 | 🔴 高 | 优先支持DXF（格式开放），DWG通过ODA处理 |
| 闭环修正中的负优化 | 🟡 中 | 引入回溯机制，修正前备份；设置迭代上限 |
| 知识库建设周期长 | 🟡 中 | 先做少量核心条款（50条），数据飞轮逐步扩充 |
| 纯开源路线的GPL合规风险 | 🔴 高 | 坚决不走纯开源路线；核心引擎用商业或MIT许可方案 |

## 七、总结

**一句话定义：** Blueprint-AI Agent = 图纸解构 + 知识库 + AI主导的合规修正闭环 + 意图驱动设计 + 全过程资料沉淀。

**核心转变：**
- 从"20个独立API" → "2个Agent闭环"
- 从"规则引擎" → "LLM+规则引擎混合"
- 从"每次分析不保留" → "全过程资料持久化"
- 从"人找工具" → "AI主动工作"

**最大壁垒：** 不是AI模型本身，而是"规范JSON库 + 图集向量库 + 过程资料库"构成的工程知识体系。这是通用大模型无法替代的，也是数据飞轮的核心。

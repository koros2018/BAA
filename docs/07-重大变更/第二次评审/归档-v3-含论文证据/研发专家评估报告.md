## 二、技术方案核心争议评审

### 争议Q1：团队规模——2人够吗？

**评估结论：2人只够做原型，不够做产品。**

| 维度 | 2人工作量估算（人周） | 说明 |
|------|--------------------|------|
| CAD引擎对接（ezdxf→ODA） | 8-12人周 | ezdxf学习曲线低但ODA SDK学习曲线极高，C++/Python混合开发 |
| AI Agent编排引擎（自研） | 6-10人周 | 工作流引擎、状态机、任务调度，自研即自造轮子 |
| 规范JSON库构建（50条） | 4-6人周 | 需建筑专家深度参与，非纯开发工作 |
| 前端（Web→Electron） | 6-8人周 | 基于现有HTML原型改造，但仍需完整的前后端联调 |
| 后端API重构 | 4-6人周 | FastAPI保留，但端点需大幅重构 |
| 测试与调试 | 4-8人周 | 图纸处理的边界情况极多 |
| 部署与运维 | 2-4人周 | Docker/私有部署/CI-CD |
| **合计** | **34-54人周** | ≈ 8.5-13.5个月 × 1人 |

**若2人并行，工期 = 17-27周（4-6.75个月），远超10周。** 第一次评审的10周估计极度乐观。

**建议：** 决策会已定2人，必须接受工期延长至4-6个月。或者将MVP范围砍到极致——仅"一张特定DXF图纸的单一规范条款审查"。

### 争议Q2：CAD引擎路线——ezdxf过渡可行吗？

**评估结论：过渡可行，但必须有明确的"ezdxf能力边界清单"。**

| ezdxf可处理 | ezdxf不可处理 |
|------------|--------------|
| 简单DXF文件（R12/R2000格式） | 二进制DWG（行业主流格式） |
| 基本图元（线、圆、文字、多段线） | 块参照、外部参照（XREF）、代理对象 |
| 图层信息读取 | 自定义对象（天正/探索者等插件生成的对象） |
| 简单几何尺寸提取 | 扩展数据（XData）、字典数据 |

**关键风险：** 如果早期合作客户使用的是"天正CAD"或"探索者"（国内90%设计院），ezdxf解析将严重失真。这种情况下，即使演示通过，实际不可用。

**建议：** 
- 开发期就购买ODA教育版（$375/年），成本极低，规避核心风险
- ezdxf仅用于验证流程逻辑，不用于客户交付
- 在技术方案中明确定义"支持的DWG版本列表"和"已知不支持的功能"

### 争议Q3：SubAgent-002的"修正策略→CAD修改"语义鸿沟

**评估结论：这是整个项目最大的技术黑洞，第一次评审对此严重低估。**

当前方案描述为：LLM生成修正策略 → 参数化模板执行 → CAD引擎修改。但实际工程中存在以下问题：

**问题1：如何从DWG中自动定位需修改的图元？**
- LLM说"楼梯宽度不够，改为1.2m"
- 但图纸中可能有10个楼梯，如何确定是哪一个？
- 需要建立"图元ID↔规范条款↔修正策略"的映射关系
- 这个映射关系目前完全缺失

**问题2：参数化模板的适用范围有限**
- 模板只能处理标准化构件（楼梯、门窗、墙体）
- 对于非标准构件（异形柱、复杂空间），模板无法覆盖
- 预计能覆盖的违规类型不超过40%

**问题3：闭环迭代的收敛性**
- 第一次评估提出了"回溯机制"和"修正评分函数"
- 但未定义评分函数的具体算法
- 没有评分函数，回溯就无法判断"哪个版本更好"

**建议：**
- MVP阶段放弃自动修正，只输出"修正建议文本+违规位置高亮"
- 将SubAgent-002推迟到V2.0，先做好SubAgent-001的精准审查
- 这在《升级简化报告》中已有提及（"不一定自动修正"），需严格执行

### 争议Q4：规范可执行化——从JSON三元组到可执行规则

**评估结论：难度被低估，但方向正确。**

当前方案中的JSON三元组示例：
```json
{
  "condition": "建筑类型=高层公共建筑 AND 楼梯类型=疏散楼梯",
  "requirement": "净宽 >= 1.2m"
}
```

**实际问题：**
- "建筑类型"和"楼梯类型"需要从图纸中自动推断，但ezdxf/ODA只能提取几何数据，不能理解语义
- 即使是人工标注，也需要先识别"这是一个楼梯"，再判断"是什么类型的楼梯"
- 当前没有"从几何到语义"的推理层

**可行的分级方案：**

| 级别 | 定义 | 实现难度 | 建议 |
|------|------|---------|------|
| L1 | 直接可提取的几何参数（如尺寸、面积、间距） | 低 | MVP首选 |
| L2 | 需组合判断的属性（如"疏散楼梯净宽"=楼梯类型+宽度提取） | 中 | V1.0 |
| L3 | 需语义理解的属性（如"建筑类型"、"防火分区"） | 高 | V2.0 |
| L4 | 需设计意图推断（如"是否满足无障碍要求"） | 极高 | V3.0 |

**建议：**
- MVP仅实现L1级别规范条款（约50条中可筛出10-15条）
- L2-L4级规范先做"人工标注属性"，再做"自动推断"
- 在JSON库中增加`difficulty_level`字段标记难度

## 四、实施计划修正建议

### 4.1 建议的MVP范围（严格执行）

```
MVP目标：在6周内实现"上传特定格式DXF → 解析 → L1级规范审查 → 输出违规报告"

不做事项：

❌ 意图驱动设计（功能二）
❌ DWG格式支持（仅DXF）
❌ 复杂规范（仅L1级几何尺寸类条款）
❌ Electron桌面（Web即可）
❌ 全生命周期管理
```

### 4.2 修正后的里程碑

| 里程碑 | 时间 | 可验证目标 |
|--------|------|-----------|
| M1: DXF解析+单规范判定 | 第4周 | 1张测试DXF，1条规范（如楼梯净宽≥1.2m），端到端跑通 |
| M2: 10条L1级规范审查 | 第8周 | 5张测试图纸，10条规范，准确率>80% |
| M3: 知识库管理界面 | 第10周 | 规范的CRUD管理、版本控制 |
| M4: 真实客户POC | 第12-16周 | 1-2家设计院试用，收集反馈 |
| M5: DWG支持（ODA集成） | 第20-24周 | 替换ezdxf，支持真实场景 |

### 4.3 自研Agent引擎 vs 使用成熟框架

| 选项 | 优点 | 缺点 |
|------|------|------|
| 自研轻量工作流 | 轻量，可控 | 需要从零实现状态管理/中断恢复/人工审批，至少6人周 |
| Temporal.io | 工业级可靠，支持长时间任务 | 学习曲线，依赖Go服务 |
| LangGraph（LangChain） | 快速原型，Agent生态 | 太重，抽象层多，调试困难 |
| **建议：Dify / FastGPT** | 开箱即用的Agent编排，支持工作流 | 定制灵活性受限 |

**推荐：使用Dify的开源版作为Agent编排引擎。** 它天然支持RAG、工具调用、工作流编排，不需要自研。2人团队不应在"造工作流引擎"上浪费宝贵的人力。

## 六、论文证据对技术争议的补充支撑（2026-06-09）

> 以下引用基于arXiv及已发表论文检索结果，为技术评估提供第三方学术证据。

### 6.1 规范JSON构建周期争议

**原争议：** 规范可执行化周期被低估3-5倍

**论文证据：**
- **LLM-FuncMapper**（Zheng et al., arXiv:2308.08728, Engineering Applications of AI, 2026）：定义了66个原子函数，LLM+adaptive prompt可将规范条款自动匹配到原子函数并组合生成可执行代码，比纯人工微调方法**高19%**。这是学术上首次将LLM用于"规范条款→可执行代码"翻译。
- **Fine-Tuning LLMs for Building Code QA**（Aqib et al., arXiv:2505.09686, 2025）：RAG+微调组合方案使规范问答准确率从53.2%提升到82.7%。

**修正影响：** 用LLM-FuncMapper方法辅助构建规范JSON库，构建周期可从预估的4-6人周压缩到**1-2人周**（含master审核时间）。AI初稿+人工审核的路线在学术上已被验证可行。

### 6.2 自研Agent框架争议

**论文证据：** 搜索到的所有相关论文（论文1-10）均采用"LLM+规则引擎"或"LLM+工作流编排"方案，没有一篇论文采用自研Agent框架。

**修正影响：** 坚持用现有框架（Dify/直接编码），不自研。学术前沿验证了这个方向是对的。

### 6.3 LLM幻觉风险

**论文证据：**
- **LLM for Building Regulation Interpretation**（Fuchs et al., 2024, arXiv:2407.19227）：LLM能理解规范语义，但精确几何参数提取和条件推理有限，必须配合规则引擎。
- **LLM-Driven Code Compliance in BIM**（Madireddy et al., 2025, arXiv:2506.12814）：LLM处理模糊条款（如"应满足相关规范要求"）时准确率下降明显，减少60%手动审查时间的前提是只处理结构化条款。

**修正影响：** LLM幻觉风险在L1级条款（尺寸/距离/面积类）上较低，在L2/L3级条款上较高。MVP阶段只做L1级条款可以控制此风险。

## 七、总结与建议

1. **MVP范围必须缩水。** 功能一（图纸合规审查）的子集 + 不做自动修正 + 仅DXF + 仅L1规范。10周内只能做这么多。
2. **立即采购ODA教育版。** $375/年，成本几乎为零，规避核心风险。
3. **不要自研工作流引擎。** 用Dify或类似开源方案，把人力省下来做核心业务逻辑。
4. **2人团队工期按4-6个月规划。** 不要用10周给master和团队虚假预期。
5. **招一个前端外包。** 或者接受Web模式，不做Electron，把前端投入降到最低。
6. **参考LLM-FuncMapper建立原子函数库。** 66个原子函数的分拆方法可直接用于BAA规范判定体系。

## 二、图纸处理——真实世界的复杂性

### 2.1 真实图纸 vs 理想图纸

第一次评审的方案假设图纸有：
- 标准图层命名（WALL, DOOR, STAIR）
- 规范的图框和比例
- 完整的尺寸标注

**真实情况是：**

| 现实问题 | 普遍性 | 对BAA的影响 |
|----------|--------|------------|
| 自定义图层名（"墙-1""墙体-主要""Q-承重墙"等10种变体） | 90%图纸 | 图层映射规则需覆盖至少50种常见变体 |
| 图纸比例混乱（1:100布局、1:50大样图混排） | 80%图纸 | 尺寸提取需乘以比例因子，但比例信息常缺失 |
| 外部参照（XREF）未绑定 | 70%图纸 | ezdxf无法处理，ODA可处理但需先绑定 |
| 天正/探索者自定义对象 | 60%图纸 | ezdxf直接解析为匿名块，丢失所有语义 |
| 图元重叠/打断 | 40%图纸 | 楼梯线被墙体线打断，AI需要"智能连接" |
| 非1:1绘制的标注 | 30%图纸 | 标注文字可能标注的是"修改后的值"而非"实际尺寸" |

**结论：** 如果用真实图纸测试，ezdxf解析的"可识别率"预计不超过40%。这意味着即使SubAgent-001逻辑正确，也无法从图纸中提取出足够的数据来运行规范判定。

### 2.2 图纸"脏数据"清洗

| 问题 | 严重度 | 处理方案 |
|------|--------|---------|
| 图层名不规范 | 🔴 高 | 建立"标准图层名↔常见变体"映射表（需收集至少50家设计院数据） |
| 比例信息缺失 | 🟡 中 | 人工标注比例，或通过图框尺寸反推 |
| 外部参照未绑定 | 🔴 高 | ODA SDK的bindXrefs功能，ezdxf不支持 |
| 自定义对象 | 🔴 高 | 必须使用ODA SDK的proxy detection功能 |

**建议：MVP阶段选择"标准化程度最高的图纸"**——由项目团队自己绘制测试图纸，确保图层规范、比例一致、无外部参照。

## 三、规范可执行化——工程视角的深度剖析

### 3.1 GB50016防火规范的可执行性分析

以GB50016-2014（2018版）防火规范为例：

| 条款类型 | 示例 | L1可执行 | L2需推断 | L3需人工 |
|---------|------|---------|---------|---------|
| 尺寸类 | 楼梯净宽≥1.2m | ✅ 可直接提取 | | |
| 距离类 | 防火间距≥6m | ✅ 可直接提取 | | |
| 面积类 | 防火分区面积≤2500㎡ | ✅ 需组合计算 | | |
| 材料类 | 防火墙耐火极限≥3h | | ✅ 需材料标注 | |
| 位置类 | 疏散出口间距≤30m | ✅ 直接测量 | | |
| 组合类 | 两个安全出口之间距离≥5m | ✅ 直接测量 | | |
| 定性类 | 疏散楼梯应采用防烟楼梯间 | | | ✅ 需设计意图 |
| 条件类 | 当设置自动灭火系统时，面积可增加1倍 | | ✅ 需系统信息 | |

**量化分析：**
- L1级别（可直接执行）：约30%的防火规范条款
- L2级别（需推断）：约40%
- L3级别（需人工）：约30%
- **MVP阶段可覆盖的条款：约15-20条（50条核心条款中的L1级别）**

### 3.2 规范条款之间的交叉引用

真实审图中最难的不是单条规范判定，而是条款之间的交叉引用和矛盾。

**示例：**
- GB50016 5.5.18要求疏散楼梯净宽≥1.2m
- GB50016 5.5.19要求人员密集场所的疏散楼梯净宽≥1.4m
- 同一建筑既是一般高层又是人员密集场所，应适用哪个条款？

**处理方案：**
- JSON库中增加`priority`字段（高优先级的覆盖低优先级）
- 增加`conflict_resolution`规则（取更严格的值为准）
- 增加`scope_overlap`标记（条款适用范围的交集）

## 四、从"AI审图"到"AI全流程"的工程路径

### 4.1 当前审图工作流

```
设计院出图 → 内部校对 → 修改 → 校对 → 送审 → 
审图中心审查 → 意见反馈 → 修改 → 复审 → 合格
```

### 4.2 BAA的定位选择

| 定位 | 优势 | 劣势 | 建议 |
|------|------|------|------|
| **替代审图中心** | 市场价值大 | 法律责任重，需资质认证 | ❌ 不可行 |
| **辅助设计院自审** | 轻量，无需资质 | 客户付费意愿低 | 🟡 可行但需验证 |
| **辅助审图中心提效** | 刚需明确，付费意愿高 | 需要渠道资源 | ✅ 推荐优先 |
| **全流程管理工具** | 差异化强 | 开发周期长 | 🟡 长期方向 |

**推荐路径：**
- V1.0：辅助设计院自审工具（轻量、快出）
- V1.5：对接审图中心API（渠道合作）
- V2.0：全流程管理（差异化壁垒）

## 七、论文证据对工程判断的补充支撑（2026-06-09）

> 以下引用基于arXiv及已发表论文检索结果，为工程评估提供第三方学术证据。

### 7.1 真实图纸处理能力争议

**原评估：** ezdxf处理真实图纸可用率<40%，需ODA教育版。

**论文证据：**
- **DoorDet**（Zhang et al., 2025, arXiv:2508.04858）：提出半自动多类别门检测管线，结合目标检测和LLM。在真实建筑平面图中实现防火门、普通门等细粒度分类检测，LLM用于清洗标注噪声。
- **Multi-View Engineering Drawings VLM**（Khan et al., 2025, ICIEA 2026）：多阶段混合框架，在真实工程图纸上达到**91.3%的语义元素识别准确率**。三阶段：视图分类→语义分割→尺寸提取与关系推理。
- **Symbol Spotting on Floor Plans**（Rezvanifar et al., 2020, arXiv:2006.00699）：深度学习符号检测**87% mAP**，用合成数据增强训练，成功处理天正/探索者自定义对象检测。

**修正：除了ODA路径，还有深度学习路径。**
- ezdxf+L1规则不够，但可以叠加DL/VLM方案做图元语义识别
- **合成数据预训练+少量真实数据微调**的迁移学习策略（论文7已验证）可直接解决"图纸脏数据"问题
- 建议路径：ezdxf解析基础几何 → 目标检测+VLM识别语义图元 → 规则引擎做合规判定
- ODA作为兜底方案（处理XREF/二进制DWG），DL/VLM方案作为核心识别引擎

### 7.2 图层映射需50家设计院数据的争议

**原评估：** 需收集至少50家设计院的图层命名规范。

**论文证据：** 论文5+6+7均采用**合成数据预训练+少量真实数据微调**策略，不需要大规模收集真实数据。

**修正：不需要50家。** 先用合成图纸训练语义理解模型，再用3-5张真实图纸做领域微调即可覆盖大部分常见变体。

### 7.3 规范可执行化难度

**原评估：** 仅30%的防火规范条款可无需人工直接执行。

**论文证据：**
- **LLM-FuncMapper**（Zheng et al., 2023/2025, arXiv:2308.08728）：定义了66个原子函数，证明规范条款可以被系统性地分解和自动化。
- **Table Comprehension in Building Codes**（Aqib et al., 2025, arXiv:2511.15575）：VLM微调后表格关键字段提取从41%提升到86%，规范中的表格数据（防火间距表等）可被有效提取。

**修正：30%的L1级别是保守估计。** 通过原子函数组合+LLM辅助解读，可执行条款比例可能提升到40-50%。但L3级别（需设计意图推断）仍需人工，这一点不变。

## 八、核心建议

1. **MVP测试图纸必须覆盖"标准化"和"真实场景"两个层次。** 先用自己的标准图纸跑通流程，再找1-2张真实设计院图纸测试，提前暴露问题。
2. **ODA作为兜底方案，DL/VLM作为核心识别引擎。** 建议优先尝试"ezdxf+目标检测+VLM"路径，如果发现天正对象无法处理再买ODA。
3. **规范库建设从L1级条款起步。** 首批15-20条尺寸/距离/面积类条款，不碰L2/L3条款。
4. **图层映射不需要50家数据。** 用合成数据预训练+3-5张真实图纸微调即可。
5. **产品免责声明必须提前准备。** AI辅助审图结果仅供参考，最终责任由设计院和注册建筑师承担。
6. **跟踪VLM在图纸理解上的进展。** 如果未来6-12个月出现能直接解析DWG+理解规范的统一模型，可能改变当前分层架构。

# BAA 项目 Code Review 检查项

> 每日开发收尾时，对当日新增/修改的代码进行自检。

## 三原则

### 1. 改动即注释
- 每次修改或新增代码时，顺手补充/更新注释
- 不要留到最后一集中补——上下文遗忘导致注释质量低

### 2. 解释 why，不解释 what
- 注释应说明业务背景、规范依据、边界条件
- 避免 `# 赋值: var`、`# 执行操作` 这类无信息量的注释
- 示例：`# 因 GB50016-5.5.18 要求楼梯净宽 ≥1.1m`

### 3. 代码审查时看注释质量
- 一段清晰的业务逻辑 > 十行 `# 执行操作`
- 注释应与代码同步更新（改代码不改注释 = 负债）

## 每日收尾检查清单

- [ ] 今日新增/修改的代码，关键逻辑是否有注释？
- [ ] 注释是否解释了 why（业务背景/规范依据/边界条件）？
- [ ] 是否存在改代码未改注释的情况？
- [ ] 注释语言是否与现有代码风格一致（中文）？
- [ ] 是否避免了批量脚本式的无意义注释？

## 覆盖率参考（非硬性指标）

| 文件类型 | 建议覆盖率 | 说明 |
|---------|-----------|------|
| 核心业务逻辑 | 70-80% | 规则复杂，需要解释 why |
| API 端点 | 40-60% | 端点功能自解释，参数有 docstring |
| 配置/数据文件 | 20-30% | 内容自解释，无需废话 |
| 简单脚本 | 30-50% | 逻辑简单，注释过多反而干扰 |

## 注释质量自检

问自己：**如果删掉这行注释，读者会误解代码吗？**

- 如果 **会** → 保留注释
- 如果 **不会** → 注释是噪音，删掉或改写
